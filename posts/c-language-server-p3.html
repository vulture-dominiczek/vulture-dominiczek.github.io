C language server: performance optimization

Here I'm continuing C language server series. In previuos part I wrote about crosslinking symbols and types between different source files. Each symbol tree is stored in hashmap by absolute path and symbols have jump_to reference to their defining type possibly in another file.

I intended to jump straight to vscode plugin implementation however I met couple of obstacles. I did some scale experiment with current code - I run indexing on kernel source code. What I found is that it would take really long time to finish parsing not mentioning crosslinking. So first I will write about performance tuning of native applications. I reached couple of interesting conclusions about what is not performant in C++. I will write about each thing in separate section.

Digression: why code navigation is lazy in VSCode

What brought me to writing language server was fact that I wasn't able to quickly get all usages of the type. In current c++ language support it just scans all the files, obviously it doesn't store any index. But when I have run my code on kernel sources I see the reason why. Some projects are really big, so it doesn't make sense to precompute any index on them, because it would take ages.

What plugins do is that they parse current source file and given the imports they can quickly fetch other files, parse them and reach the definition. So everyhing is on demand. The problem is of course finding all the usages because that would require aforementioned index.

Some IDE's however tend to take opposite approach and put every file in index. For example as a Java programmer I am used to IntelliJ and for big projects it takes at most 30 seconds to index. This makes me wonder how is it possible. To beat parsing 100 files per second is almost impossible on commodity hardware, so either they employ smart heuristics, or they precompute the indexes (for standard lib, for ex.) and ship them with the IDE. Or maybe it is a mix of them... Some day maybe I will look at the source code to see.

I was trying to make the most of my PC. I used valgrind with it's ui kcachegrind to debug stuff. The pattern was simple: I sampled the running application on kernel sources. Then checked which method takes most time percentage and fixed that. So to reach optimal performance I had to polish some part's of the code and they are described in secions below:

C-string to String coversions

To my utmost surprise 


std::basic_string<CharT,Traits,Allocator>::length

took the most of the time. Tree sitter is in C so it's API passes zero terminated strings. Originally I used std::string because it has == operator overloaded. And string (I assume) stores length component as well. So when string is constructed from C-string it must iterate through all chars to get the length.

Counting around 100 symbols/definitions/references per file for thousand files it would give a lot of conversions. This may seem small but each identifier is several characters so it all sums up eventually. So to give it a performance boost I had to give up string's == to strcmp and this eliminated this occurence.

Move semantics

This wasn't an issue here but thought I would mention it briefly. Some time ago in C++11 there was new construct introduced. The reason was that the objects managing some external objects on the heap were previously duplicated by copy constructor and copying takes time. To gain performance we can instead switch reference from right object to left object in the constructor and no copying takes place.

So this achieved by having 2 types of objects:

  • lvalue - it has assigned name and address
  • rvalue - it is temporary object for which you can't dereference

Now if you switch lvalue with rvalue in the move constructor you can move rvalues state to lvalue and rvalue object is temporary either way so we won't use it any more.

By using std::move we cast the objects reference to rvalue and by such we can use it in move constructor, but can't use it anymore in any other place.

Smart pointer std::unique_ptr already uses this semantics and can be used C++11 upwards.

Pairs and tuples

My first idea was to hold hashmap key as as tuple and it was a mistake. So here is my lesson of the day, structs are fast and tuples are slow. Any time you want to access tuples component you have to invoke templated method instead reach to the field. On the margin it is terribly inconvenient either way.

So I rewrote the key to struct with == operator overloaded and specialized hash template in std namespace as required. It had a huge speedup.

Unnecessary vectors

So another place in found inefficiency was that while I was parsing the CST I stored temporary vector for every node. Maybe it is not big deal but when the code is invoked hundreds of thousands of times it all sums up.

So what I did I somehow mimicked newtype pattern from Rust | Haskell. I wrapped C structure TSNode in a class and this helped me to implement methods that delagated to ts_node_child_count and ts_node_child passing this structure but making it like a call on object.

Regex

First thing with regex is that it needs to be compiled and this takes time. This took 80% of remaining time in valgrind. First I tried to replace C++ regex with Google's Re2, but still performance problems persisted. On the margin, Re2 is definitely nicer to use.

Regex is kind of swiss army knife made for everything. This complexity costs performance penalty and it was not needed for my use case anyways. I only needed to normalize references to the "dot separated string" so I extended the build_stack_graph for parsing CST with expressions. If it was field expression it recursed prepending field with dot.

Conclusion

That's it. I knew that I optimized away most of stuff when Tree Sitters API's percentage in performance samples was around 80%. So after doing all this I reached performance of indexing 50 files/second. It is quite nice. Linux kernel is medium sized project and it takes around 1,5 minute to index it. It would be nice for bigger projects to save index to file when indexing is done but I will pass it for now.
