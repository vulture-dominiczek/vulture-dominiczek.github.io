Kafka on Kubernetes: exercise

Some projects make a heavy use of Kafka while also being on Kubernetes. The two don't go well together that's for sure. So sometimes it turns out companies spawn only one instance of Kafka which is suboptimal solution. I found it interesting topic to write how to setup cluster of Kafkas on k8s.

Kafka clustering

Up to version 2.8 Kafka relied only on Zookeeper for managing it's state and leader election. In a Kafka cluster producers write to leader and it's partitions get replicated by topic to other nodes. The leader has controller that manages this stuff. And Zookeeper oversees this communication and maintains state.

As Confluent team states the replication with zookeeper became increasingly complex, so they started looking for a way to solve it. In version 2.8 KRaft mode was introduced and it completely goes without Zookeeper. All the synchronization happens just between the brokers. It is said that it is much faster with big amount of topics.

We will try classic approach with zookeeper because Kraft is not advised for production yet.

Setting up the cluster

For Kafka brokers to work we essentially need to set up two config values:

  • advertised.listeners - any instance will broadcast this addresses to clients as reachable brokers
  • listeners - brokers will listen on these addresses for replication updates
  • zookeeper.connect - addresses of zookeper cluster nodes

Now with Kubernetes we have a chicken and egg problem that we don't know cluster ip's that will get assigned to these pods.

The second problem is that we want pods to have identity. If broker 1 fails we want it recover as broker 1 not for ex. broker 4. And broker 1 should have same volume mounts as before. We would also like to be able to migrate pods between the nodes (if one fails).

How do we solve these?

For the first, we will use headless service. The trick is that it enables communication between the pods (but only internally) but it disables the loadbalancing. This way each broker can talk with another by "central hub". And dns record for the headless service returns all of the pod ips.

For the second there is quite old addition to the Kubernetes: StatefulSet. Each pod has identity and volume mounts stay the same regardless of fortune. Moreover pods get consistent names like "podName-number". We will do that for Kafka only.

Let's assume that we don't need expose kafka outside kubernetes (many times this is the case), then with the above two we are done.

So first we are creating "base image" for Kafka.


FROM debian:buster-slim

RUN apt-get update

RUN apt-get install -y wget openjdk-11-jdk gettext-base

RUN wget https://dlcdn.apache.org/kafka/3.1.0/kafka_2.13-3.1.0.tgz

RUN tar -xvf kafka_2.13-3.1.0.tgz

RUN mv kafka_2.13-3.1.0 /kafka
Zookeeper

So first we need to setup an image:


FROM kafka-base:1.0

WORKDIR /kafka 

RUN mkdir /tmp/zookeeper

COPY ./zookeeper-0.properties .
COPY ./zookeeper-1.properties .
COPY ./zookeeper-2.properties .

CMD export MY_ID="${POD_NAME##*-}" && echo $MY_ID && echo $(($MY_ID+1)) > /tmp/zookeeper/myid && bin/zookeeper-server-start.sh ./zookeeper-$MY_ID.properties

We inject env variable POD_NAME, derive number from it and then assign each pod separate configuration. zookeeper-0.properties looks like so:


dataDir=/tmp/zookeeper

maxClientCnxns=0
admin.enableServer=false

server.1=0.0.0.0:2888:3888
server.2=zookeeper-statefulset-1.zookeeper-headless.default.svc.cluster.local:2888:3888
server.3=zookeeper-statefulset-2.zookeeper-headless.default.svc.cluster.local:2888:3888

tickTime=2000
initLimit=5
syncLimit=2
electionPortBindRetry=100

we inject id so we know that the first server should bind to generic address to listen to. By this configuration we enabled cluster mode. Notice how we use DNS instead of IPs thanks to headless service. The generic address of the pod is "pod-name"."service-name".default.svc.cluster.local.

Now comes the time for Kubernetes setup:


apiVersion: v1
kind: Service
metadata:
  name: zookeeper-headless
  labels:
    app: zookeeper-headless
spec:
  ports:
  - port: 2888
    name: hub
  - port: 3888
    name: hub2
  - port: 2181
    name: hub3
  clusterIP: None
  selector:
    app: zookeeper

---

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: zookeeper-statefulset
spec:
  serviceName: "zookeeper-headless"
  replicas: 3
  selector:
    matchLabels:
      app: zookeeper
  template:
    metadata:
      labels:
        app: zookeeper
    spec:
      containers:
      - name: zookeeper-container
        image: zookeeper:1.0
        imagePullPolicy: Never
        ports:
        - containerPort: 2888
        - containerPort: 3888
        - containerPort: 2181
        env:
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name

So we exec:


kubectl create -f kafka.yaml

and check the logs:


kubectl logs --tail=200 -l app=zookeeper

The output assures that the cluster is running:


[2022-04-15 09:21:04,017] INFO FOLLOWING - LEADER ELECTION TOOK - 226 MS (org.apache.zookeeper.server.quorum.Learner)

So we are done with the zookeeper now it's time for kafka:

Kafka

For Kafka we also prepare a Dockerfile and properties but this time with a slightly different technique. We will substitute variable in properties fiele with envsubst command. Dockerfile:


FROM kafka-base:1.0

WORKDIR /kafka 

COPY server.properties .

RUN cat server.properties

RUN mkdir /var/kafka

CMD export MY_ID="${BROKER_ID##*-}" && envsubst < server.properties > server-subst.properties &&  cat server-subst.properties && bin/kafka-server-start.sh server-subst.properties

And properties file:


log.dir = /var/kafka

listener.security.protocol.map = INTERNAL:PLAINTEXT

broker.id = $MY_ID

zookeeper.connect = zookeeper-headless.default.svc.cluster.local:2181

listeners = INTERNAL://0.0.0.0:9092

advertised.listeners = INTERNAL://kafka-statefulset-$MY_ID.kafka-headless.default.svc.cluster.local:9092

inter.broker.listener.name = INTERNAL

There we injected Id computed in command. We are passing headless service as zookeeper connection details and pass our headless service for kafka as cluster details.

Kubernetes configuration file:


apiVersion: v1
kind: Service
metadata:
  name: kafka-headless
  labels:
    app: kafka-headless
spec:
  ports:
  - port: 9092
    name: hub
  clusterIP: None
  selector:
    app: kafka

---

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: kafka-statefulset
spec:
  serviceName: "kafka-headless"
  replicas: 3
  selector:
    matchLabels:
      app: kafka
  template:
    metadata:
      labels:
        app: kafka
    spec:
      containers:
      - name: kafka-container
        image: kafka-broker:1.0
        imagePullPolicy: Never
        ports:
        - containerPort: 9092
        env:
        - name: BROKER_ID
          valueFrom:
            fieldRef:
              fieldPath: metadata.name

This is basically the same configration as with zookeeper so no explanation needed i think.

Testing

Now let's see the queueing in action.

Let's run a pod with our base image.


kubectl run -i --tty test --image=kafka-base:1.0

Now according to this quickstart we create topic:


root@producer:/# /kafka/bin/kafka-topics.sh --create --topic chat --bootstrap-server kafka-headless.default.svc.cluster.local:9092
Created topic chat.

Then produce some strings:


root@producer:/# /kafka/bin/kafka-console-producer.sh --topic chat --bootstrap-server kafka-headless.default.svc.cluster.local:9092
>Hello
>It's a beautiful day
>^C

And check it works:


root@producer:/# /kafka/bin/kafka-console-consumer.sh --from-beginning --topic chat --bootstrap-server kafka-headless.default.svc.cluster.local:9092

Hello
It's a beautiful day
^CProcessed a total of 2 messages

Conclusion

You can use Kuberenetes StatefulSet and Headless Service to setup Zookeeper and Kafka clusters. The more the nodes the more resilient the cluster is but this negatively impacts latency and throughput as well.
