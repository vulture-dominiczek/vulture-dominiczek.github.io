Languages, evolution and compatibility

Today I watched amazing talk by Brian Goetz on how is it to maintain a language. This video spurred lots of thoughts in my head and they still keep spinning so I thought I will write a post. The languages come and go and they evolve. What causes them to change?

Evolutionary model

In a video there was an analogy given as the problems are prey and languages are predators. The language is either fit for the problem in which case the community thrives or not and it loses popularity and dies out eventually.

Mathematic models always predated the development of algorithms and infrastructure. What is called new paradigm is actually incarnation of some theory/model invented long time ago. So problems as such are pretty constant, but the need what to solve changes.

What really keeps changing stems from technological advance:

  • people started using facebook so there is a need for graph algorithms
  • OO and functional programming got traction, because they are closer to real life
  • cheap computers and specialization got as to the cloud
  • the data got overwhelmingly big, so we need to process them in parallel

This obviously had to result either in birth of new languages or evolution of the older. However there is a limit in how far the evolution can go for particular language. In the beginning some assumptions were made and this narrowed down the range of possibilities for next language features. I have never seen a language fully rewritten to contain completely different features. Sometimes there is possiblity to fix this and that but the core is set in stone.

You can't introduce feature in a range of small steps over the years, because planning the undertaking wouldn't fit in anyones head for sure and it would take to long. It's better to create new language.

Free market

Complementary view of programming languages would be as they are the bunch of shops, competing with each other for developers popularity. Some are have big impact on the programming world. Some are smaller but more specialized. What is constant is that the language itself (it's maintainers) want to attract as much of developers as possible.

If one language introduces new feature in matter of months/years that proves valuable everybody else jumps on the wagon and implement it's for themselves. Scala capitalized on Java's weaknesses and got type system and functional programming really well. But now Java is catching up and merges the features that set it apart. And Scala doesn't have big industry adoption and no company backs it, which will make it remain niche and possibly forgotten.

Programmer's are very fashionable people. And fashions come and go also, and so the features in languages. These days functional programming is very popular. Mainly because of lack of side effects which is good for computation at scale. But in future we can have different kinds of architectures (quantum computing) which may reduce big distributed architectures to monolith once again.

Conclusion

So the evolution analogy works for small to medium features until it is practical to incorporate it in current design. Whenever there is paradigm shift new langauges spring up. And languages features are largely influenced by popularity among developers because every language tries to maximize developers happiness. There is constant push and pull between compliance to the existing codebase, popularity and new trends and specializing in being one tool that does solve the problem really well.
