C language server: VSCode Plugin

This is the last post I'm planning to write in the series. The server has been implemented and it is working to reasonable extent. Now what is left to do is Visual Studio Code plugin and communication between the two. The whole code is available here if you want to know the nitty-gritty details.

First some picture:

This is the outline of the system working. I was wondering what is simple and efficient way to implement communication. I picked grpc first enticed by performance. Since the payloads would be binary the latency should be minimal. But then I realized my payloads are rather small and it would be overkill for this purpose and setup is rather hard in C++. Besides it don't like the idea of regenerating classes each time I change schema.

JSON-RPC is the way VSCode guys chose to standarize on. But then I would be tided to the LSP/LSI spec and would need to implement all the stuff I don't need and don't have time to do. I was left with implementing communication all by myself.

But is there a lot of work to be done? Not really... I went with very simple schema: plugin spawns a server process and acquires it's standard input and output. Then the communication is simply by exchanging lines between server and client. And each line is stringified JSON object. Request and reponse have command field indicating which action should be taken by server. Server responds with payload correlated with command name again. The solution is very minimal and easy and I like it.

The plugin

The API is organized as follows. First there are 2 categories of plugins: normal and languages support. To be honest I was expecting something different for language integration. It is very tightly built in the editor and you are dependant on the specs mentioned.

Other plugins have more freedom but not everything can be done easily. In theory any action could be done by commands but sometimes you also have to read editors contents. Then there are objects like window.activeTextEditor from which you can fetch the file contents.

So I abandoned the idea of implementing language integration the official way. The way I did it instead is registering 3 commands: index, resolve definition and find usages and binding them to key shortcuts. Indexing and finding definitions went smooth because they didn't need custom gui do display results. Finding usages required displaying page with list of files and line locations.

I will only paste a small part how implementation of the plugin looks like.


else if (data.command == 'resolve') {
    if (data.status == "not_found") {
        vscode.window.showInformationMessage('Not found!');
    }
    else {
        await vscode.commands.executeCommand("vscode.open", vscode.Uri.parse(data.coordinate.path));
        let editor = vscode.window.activeTextEditor;
        let range = editor.document.lineAt(data.coordinate.line).range;
        editor.selection = new vscode.Selection(range.start, range.end);
        editor.revealRange(range);

    }
}

Here you can see how you can navigate to definition on reference found. First you open the absolute path, luckily if file is inside opened directory the file merges into already opened. Then there is no simple way to move cursor. Instead you can mock the behaviour by selecting a given line with editor.revealRange(range)

This was for finding definition. Other way around for displaying all usages, the only way I found that didn't require lots of work was to display html page with links by vscode.window.createWebviewPanel. So basically I used string interpolation to craft html, put it inside panel and display it to the side. But then it would be nice for the links to be clickable, which is not supported currently. While searching the internet I found the github issue however in which developers of VSCode showed a hack how to do this. VSCode provides in browser api to post messages to the node engine. Then, server side you can parse it and open the link from there.

Here is the code:


const vscode = acquireVsCodeApi();

for (const link of document.querySelectorAll('a[href^="c-lang-ref:"]')) {
    link.addEventListener('click', () => {
        vscode.postMessage({
            command: "open",
            link: link.getAttribute('href').replace('c-lang-ref', 'file'),
        });
    });
}

You put this code in script tag inside panels html. Then you tweak panels options to enable script execution. And then you parse message from inside the plugin and open the document linked.

C++ part

Now very shortly about the sending the messages. I found very nice json implementation for C++. It acts like a hash map that makes it really easy to work with. It is used like so (for reading):


 void loop()
    {
        while (true)
        {
            std::string line;

            std::getline(std::cin, line);

            json parsed = json::parse(line);

            switch (hash(parsed["command"].get<string>().c_str()))
            {
            case hash("stop"):
                exit(0);
            case hash("index"):
                do_index(parsed["payload"]);
                break;
            case hash("resolve"):
                resolve(parsed["payload"]);
                break;
            case hash("find_usages"):
                find_usages(parsed["payload"]);
                break;
            case hash("debug_print_tree"):
                debug_print_tree(parsed["payload"]);
                break;
            default:
                std::cout << line << std::endl;
            }
        }
    }

The server never ends. It's simply killed on close by the plugin. In each iteration it waits for a line on stdin and parses the JSON. C++ still doesn't have switch on strings so you have to do a trick and hash it to integer. Anyways, next the field "command" dispatches the action to be taken.
