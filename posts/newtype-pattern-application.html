One use of newtype pattern

Recently I was working on tree algorithm that was rather complex. If this was not enough there were different types of nodes, which complicated matters even more by ugly if-else blocks. I thought for a while how can get rid of these and aha! Encapsulation. Instantly I thought of newtype pattern not widely know among programmers. What follows is how you can use it to reduce complexity a bit.

Prelude

I already wrote about it couple of posts before but I will briefly refresh it. In Haskell you have this construct by which you can encapsulate existing type in a new wrapper. Then this becomes compeletely different type and somewhat you hide what you already wrote for old type behind this wrapper.

It becomes even more interesting when combined with OOP in Java. Let's say you have a tree with different kinds of nodes.


    public class Root {
        public final List<InternalNode> children;

        public Root(){
            this.children = new ArrayList<>();
        }

        public void addChild(InternalNode node){
            this.children.add(node);
        }
    }

    public class InternalNode {
        public final List<InternalNode> children;
        public final List<Leaf> attributes;

        public InternalNode() {
            this.children = new ArrayList<>();
            this.attributes = new ArrayList<>();
        }

        public void addChild(InternalNode node){
            this.children.add(node);
        }

        public void addAttribute(Leaf att){
            this.attributes.add(att);
        }
    }

    public class Leaf {
        public final Object value;

        public Leaf(Object value) {
            this.value = value;
        }
    }

Imagine that old codes requirement was that each type of node has different kinds of logic attached or this logic is written with if's. You can't touch this structure, because either it's too much work or you don't control external jar.

Now imagine how processing this tree would look like...


    public static void walkTree(Root root, InternalNode internalNode, Leaf attribute){
        if(root != null){
            System.out.println("Visited Root");
            root.children.forEach(in -> walkTree(null, in, null));
        }
        else if(internalNode != null){
            System.out.println("Visited InternalNode");
            internalNode.children.forEach(in -> walkTree(null, in, null));
            internalNode.attributes.forEach(att -> walkTree(null, null, att));
        }
        else {
            System.out.println("Visited attribute, value="+attribute.value);
        }
    }

From the first sight you can't see what the code is doing, it's too blurred by the if's.

Newtype

But you can write your new logic in a different way. So to simplify this let's take those if's away and put them somewhere else. But where? How about creating one universal node type that aggregates all three types of given nodes in itself. 


    public class UniversalNode {
        public enum NodeType {ROOT, INTERNAL, LEAF};

        public final NodeType type;
        private Root root;
        private InternalNode internalNode;
        private Leaf leaf;

        public UniversalNode(Root root){
            this.root = root;
            this.type = NodeType.ROOT;
        }

        public UniversalNode(InternalNode internal){
            this.internalNode = internal;
            this.type = NodeType.INTERNAL;
        }

        public UniversalNode(Leaf leaf){
            this.leaf = leaf;
            this.type = NodeType.LEAF;
        }
    }

This doesn't buy us anything yet. But having this structure we can treat all 3 different types of nodes as one. Just write one universal method doing something and hide the branching in there. First let's implement logic of adding a child to UniversalNode.


     public void addChild(UniversalNode other){
            if(type == NodeType.ROOT){
                if(other.type == NodeType.ROOT || other.type == NodeType.LEAF){
                    throw new IllegalArgumentException("Doesn't conform to tree structure");
                }
                else {
                    this.root.addChild(other.internalNode);
                }
            }
            if(type == NodeType.INTERNAL){
                if(other.type == NodeType.ROOT ){
                    throw new IllegalArgumentException("Doesn't conform to tree structure");
                }
                else if (other.type == NodeType.LEAF){
                    this.internalNode.addAttribute(other.leaf);
                }
                else {
                    this.internalNode.addChild(other.internalNode);
                }
            }
            if(type == NodeType.LEAF){
                throw new IllegalArgumentException("Doesn't conform to tree structure");
            }
        }

By doing this now we have a tree of only one type of Node. Let's go further and implement other methods:


       public List<UniversalNode> getChildren(){
            if(type == NodeType.LEAF){
                return List.of();
            }
            else if(type == NodeType.INTERNAL){
                var lst = this.internalNode.children
                        .stream().map(UniversalNode::new).toList();
                lst.addAll(this.internalNode.attributes.stream().map(UniversalNode::new).toList());
                return lst;
            }
            else {
                return this.root.children.stream().map(UniversalNode::new).toList();
            }
        }
        
        public void printSelf(){
            if(type == NodeType.LEAF){
                System.out.println("Visited Leaf");
            }
            else if(type == NodeType.INTERNAL){
                System.out.println("Visited InternalNode");
            }
            else {
                System.out.println("Visited Root");
            }
        }

Now take a look how you can rewrite tree walking to much simpler form.


       public static void walkTree(UniversalNode un){
        un.printSelf();
        for(UniversalNode u : un.getChildren()){
            walkTree(u);
        }
    }

Tree

From this an interesting structure emerges. In terms of operations like addChild, setParent etc we are operating on an outer nodes. And tree is visible from the outside world as built out of single type of node. But in those methods we are maintaining internal structure of "nested" nodes so we have 2 trees one inside another. When we are done we just get the internal node of the root and get the old structure back. This is pretty elegant in my opinion.

Conclusion

You maybe thinking meh what a big deal... This example is very simple, but sometimes you can happen to be dealing with very algorithmetically intensive and exhausing work. You want to keep this part as simple as possible to facilitate learning for newcomers for ex. And by moving if's to universal type the rewrite of the method was possible to much cleaner form. I hope this makes sense, see you around.
