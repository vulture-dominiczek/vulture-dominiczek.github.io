k8s on GCP: a walkthrough

Kubernetes has been mainstream for some time already. And everybody seems to have already moved to cloud. At work I had a bit of GCP and that time found devops work pretty cool. This post is a challenge that I went through: setting up Kubernetes on bare VMs.

It is like a tutorial so you have to read it from top to bottom. And configured k8s cluster is a result. It touches on a lot of things regarding cluster administration and GCP in general.

But maybe before we dive straight in let's see if this work would be useful at all. Below is pros and cons of setting up cluster by yourself versus cluster that is provisioned for you by GCP.

ManagedUnmanaged
Control plane is fully managed and configuration is impossible or serverly limitedFull control of control plane configuration any administrative actions can be taken
Number of worker nodes scales automatically with increase in loadManual administrative work is needed to scale
Nodes limited to running on OS images given by cloud providerPossiblity to bake your own OS images
Nodes run on VMs which state is completely hidden from devopsFull control over nodes' VMs
Integrated logging and monitoringManual setup of logging and monitoring
Filesystem not persistent over restartsPossiblity to backup and setup persistent storage
Automatic upgrades and autohealingManual maintanance of individual nodes or custom automation
Authorization and Authentication by Google IAMAuthorization and Authentication by bring your own certificates, passwords, proxies

Architecture design

Before laying out the components it's good to know what purpose our cluster will serve. These are the constraints:

  • if one physical location fails this shouldn't affect both control plane and workers
  • workloads are small and only for demonstration purposes
  • setup should be maximally simple

There are 2 approaches to setup control plane: one colocates etcd instances with other Kubernetes components (easier), other externalizes etcd to separate cluster (harder). Google Cloud Platform for most of the resources offers at most regional resiliency. Zone are servers in one building in some city of your choosing and region is couple of such buildings in this city. Global resiliency is achieved by having clusters of resources duplicated on each continent, with switching clients to the closest cluster available. It's because latencies across continents are simply to big to have cluster distributed around the world.

Let's have three control plane nodes in regional configuration: each in separate zone. And workloads are small so 3 workers is more than enough. This way high availability will be achieved at minimal cost. Control plane will be side by side with etcd to omit separate etcd cluster setup. Like in this diagram:

Setup and tools

This is a lot of work and some time ago it had to be done manually. These days however infrastracture as code is taking over the world. You maintain declarative definition and automation tools handle creation, deletion and updates of needed resources. By now, the most popular tool is Terraform and I used it in this post.

To start coding Terraform scripts you need to setup couple of GCP related things:

  • create GCP project and enable billing
  • enable Compute API from the Console
  • download and install gcloud SDK
  • create service account with editor permissions and download the key
  • install Terraform

Terraform providers and utilities

Terraform structures project as a folder with script and data files. All of Terraform code goes to main.tf file. You can extract variables to variables.tf file, but I won't use that feature as the script will be really simple. In script file you define resources which you want to setup (not only) on the cloud. Running the script triggers build of Resource Graph: dependencies are ordered and non-dependant resources are free to be provisioned in parallel. Before actual actions happen Terraform displays Execution Plan that lists all the operations to be done. Steps already finished and resource modifications are persisted in special file tfstate so that only incremental changes are applied. Once you are happy with the script you run terraform apply to provision infrastructure or terraform destroy to tear it down. This is the first part of main.tf:


terraform {                                          // 1
  required_providers {
    google = {
      source = "hashicorp/google"
      version = "3.5.0"
    }
  }
}

provider "google" {                                 // 2
  credentials = file("service-account-key.json")

  project = "dg-kubernetes-2"
  region  = "europe-west3"
  zone    = "europe-west3-a"
}

provider "google-beta" {                            // 3
  credentials = file("service-account-key.json")

  project = "dg-kubernetes-2"
  region  = "europe-west3"
  zone    = "europe-west3-a"
}

locals {                                            // 4
  control_ips = ["10.1.0.10", "10.1.0.11", "10.1.0.12"]
  worker_ips = ["10.1.0.13", "10.1.0.14", "10.1.0.15"]
  control_zones = ["europe-west3-a","europe-west3-b","europe-west3-c"]
  worker_zones = ["europe-west3-a","europe-west3-b","europe-west3-c"]
}
  • Terraform interacts with the cloud and servers by plugins called providers. This block shows which providers are required in the script.
  • This is the configuration of provider. Similar to gcloud SDK, plugin requires project, region and zone labels. Next, GCP Service Account's JSON key (put in our project directory) contents are read.
  • GCP provider has couple of resources in beta stage and they reside in separate plugin. It is configured the same way.
  • Terraform allows to keep local variables in one place.

Configuring VPC network

Kubernetes cluster requires two network interfaces. First one serves communication on control plane, but second is for virtual, inside-kubernetes network and assigns each pod unique IP only inside cluster. We can configure it in two ways. An Alias IP is a secondary interface for each VM that fits this use case nicely. Or second option is to enable VMs to forward misrouted packets to each other. Google Cloud Platform's VPC Network is a global resource. It is built from Subnets that are scoped to region. They are basically IP ranges from which you can assign individual IPs to Virtual Machines (inside the given region). Yet thats not everything - you also have to enable receiving or sending traffic by security rules either by Routes or Firewall Rules. Routes deal with outbound traffic from VMs. Firewall Rules enable/disable inter-VM traffic.

We configure VMs to have IPs from single subnet. Next inter-VM traffic by TCP, UDP and ICMP is enabled. Both ranges primary and inside kubernetes are taken into account. Kubernetes API Server talks to kubectl on TCP 6443, so traffic from default gateway (0.0.0.0) on this port has also to be enabled. This gives following piece of code:


resource "google_compute_network" "vpc_network" {         
  name = "kubernetes-network"
}

resource "google_compute_subnetwork" "vpc_subnet" {       
  name          = "kubernetes-subnet"
  ip_cidr_range = "10.1.0.0/24"
  region        = "europe-west3"
  network       = google_compute_network.vpc_network.id
}

resource "google_compute_firewall" "fr_internal" {        // 1
  name    = "kubernetes-fr-internal"
  network = google_compute_network.vpc_network.name

  allow {
    protocol = "icmp"
  }

  allow {
      protocol = "tcp"
  }

  allow {
      protocol = "udp"
  }

  
  source_ranges = [ "10.1.0.0/24", "10.2.0.0/16" ]
}

resource "google_compute_firewall" "fr_external" {        // 2
  name    = "kubernetes-fr-external"
  network = google_compute_network.vpc_network.name


  allow {
    protocol = "icmp"
  }

  allow {
      protocol = "tcp"
      ports = [ "22" , "6443" ]
  }

  source_ranges = ["0.0.0.0/0"]
}

resource "google_compute_firewall" "fr_external2" {       // 3
  name    = "kubernetes-fr-external2"
  network = google_compute_network.vpc_network.name
  direction = "EGRESS"

  allow {
      protocol = "tcp"
      ports = ["6443" ]
  }

  destination_ranges = ["0.0.0.0/0"]
}
  • This part enables inter-VM communication.
  • We have to enable HTTPS for kubectl and SSH to log into machine.
  • Required for health checks, more on this later.

Configuring Virtual Machines on GCP

The essence of the cluster are of course Virtual Machines. They are virtualized computing resources with configurable number of vCPUs (machine type). They are scoped to single zones. Any VM has primary hard drive called boot disk on which there is operating system present. Each cloud provider maintains range of OS images, both Windows and Linux from which you can init this disk. Each VM has also IP on its primary network interface.

Continuing configuration minimal number resources used by VMs is set up. Similar, but not the same are definitions of control plane and worker instances. Kubernetes has to be installed on each machine separately. To save yourself some work there is possilibity to run common script on init of machine by metascript.


resource "google_compute_instance" "control_plane_instances" {        // 1
  name         = "control-plane-instance${count.index}"
  machine_type = "e2-standard-2"
  zone = local.control_zones[count.index]
  count = 3                                                           // 2

  boot_disk {                                                         // 3
    initialize_params {
      image = "ubuntu-os-cloud/ubuntu-1804-lts"
      size = 128
    }
  }

  network_interface {                                                 // 4
    network = google_compute_network.vpc_network.name
    subnetwork = google_compute_subnetwork.vpc_subnet.name
    network_ip = local.control_ips[count.index]
    access_config {
    }
  }

  can_ip_forward = true                                               // 5

  metadata_startup_script = "${file("init.sh")}"                      // 6
}

resource "google_compute_instance" "worker_instances" {               // 7
  name         = "worker-instance${count.index}"
  machine_type = "e2-standard-2"
  zone = local.worker_zones[count.index]
  count = 3

  boot_disk {
    initialize_params {
      image = "ubuntu-os-cloud/ubuntu-1804-lts"
      size = 128
    }
  }

  network_interface {
    network = google_compute_network.vpc_network.name
    subnetwork = google_compute_subnetwork.vpc_subnet.name
    network_ip = local.worker_ips[count.index]
    access_config {
    }
  }

  can_ip_forward = true

  metadata_startup_script = "${file("init.sh")}"
}
}
  • The block provides configuration for control plane instances.
  • count is a special attribute in Terraform - it triggers a loop in which enclosing block is repeated. Iterated values are substituted by count.index. Recall that we declared array of IPs in subnet range for control plane. Now they are filled in network section.
  • Boot disk initial parameters.
  • VMs connect to network and subnet created earlier.
  • Very important, it allows VMs to forward packets to other VMs, allowing Kuberenetes nested network to function.
  • Pre-provisioning script to run on VMs boot.
  • Similar configuration is for worker nodes with couple of exceptions.

Since OS is linux distribution, provisioning script is just bash script. The contents of init.sh are:


sudo apt-get update

sudo apt-get install -y \
    apt-transport-https \
    ca-certificates \
    curl \
    gnupg \
    lsb-release

curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg
curl -fsSLo /usr/share/keyrings/kubernetes-archive-keyring.gpg \
https://packages.cloud.google.com/apt/doc/apt-key.gpg

echo   "deb [arch=amd64 signed-by=/usr/share/keyrings/docker-archive-keyring.gpg]  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

echo "deb [signed-by=/usr/share/keyrings/kubernetes-archive-keyring.gpg] https://apt.kubernetes.io/ kubernetes-xenial main" | sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt-get update

sudo apt-get install -y docker-ce docker-ce-cli containerd.io kubelet kubeadm kubectl

sudo apt-mark hold kubelet kubeadm kubectl

This script installs docker (running containers), kubelets (network proxy for Kubernetes) and kubeadm (administration tool).

GCP load balancer

In this section frontend for control plane is configured. If you remember there are three control plane nodes listening on port 6443 for connection from kubectl. Google Cloud Platform provides load balancer, but to connect it to VMs they have to be wrapped in Instance Groups. Moreover Health Check needs to be configured to constantly ping those VMs. On failed pings from VM, load balancer will redirect traffic to healthy ones. Kubernetes operates in TCP/IP layer, so external TCP load balancer has to be configured (as opposed to HTTP/HTTPS load balancer).


resource "google_compute_instance_group" "control_plane_igs" {                          // 1
  name        = "control-plane-instance-group${count.index}"
  count = 3
  zone = local.control_zones[count.index]

  named_port {
    port = 6443
    name = "control"
  }
 
  instances = [google_compute_instance.control_plane_instances[count.index].self_link]
}

resource "google_compute_region_backend_service" "control_plane_backend" {            // 2
  region = "europe-west3"
  name      = "control-plane-backend"
  protocol  = "TCP"
  provider = google-beta

  backend {
    group = google_compute_instance_group.control_plane_igs[0].id
  }
  
  backend {
    group = google_compute_instance_group.control_plane_igs[1].id
    
  }

   backend {
    group = google_compute_instance_group.control_plane_igs[2].id
    
  }

  port_name = "control"

  health_checks = [
    google_compute_region_health_check.tcp_health_check.id
  ]

  load_balancing_scheme = "EXTERNAL"
}

resource "google_compute_region_health_check" "tcp_health_check" {                // 3
  name     = "tcp-region-health-check"

  timeout_sec        = 1
  check_interval_sec = 1

  tcp_health_check {
    port = "6443"
  }
}

resource "google_compute_forwarding_rule" "default" {                             // 4
  name                  = "control-plane-forwarding-rule"
  region                = "europe-west3"
  port_range            = 6443
  backend_service       = google_compute_region_backend_service.control_plane_backend.id
  
  load_balancing_scheme = "EXTERNAL"
  provider = google-beta
}
  • To connect VMs to backend instance groups need to be created. Here each has only one VM.
  • Load balancer connects traffic to backends, each of them points to Instance Group.
  • Health check.
  • Finally load balancer is defined.

At this point you run terraform apply and wait until completion. After completion write down load balancer's external address from terraform.tfstate

Provisioning with kubeadm

Now the infrastructure is ready on Google Cloud Platform. Some manual work is required to form a cluster, install pod network and setup connection for admin. Important! Two control plane nodes must be disabled temporarily to not confuse load balancer while setting up control plane. This can be done from the GCP console. Lets log in by SSH into on VM.


gcloud config set compute/region europe-west3
gcloud config set compute/zone europe-west3-a
gcloud compute ssh control-plane-instance0

...and init kubernetes control plane node:


sudo kubeadm init --control-plane-endpoint=<lb ip>:6443 \
  --pod-network-cidr=10.2.0.0/16 --upload-certs

It begs for some explanation: we allowed second CIDR range in firewall. Now it is provided to kubeadm to form cluster connectivity. Then external IP for cluster needs to be specified (or DNS entry). Upload certs flag makes sure new control plane nodes receive certificate. This certificate is signed for all control plane nodes and load balancer's IP. The long output provides command we have to run on other two nodes.


Your Kubernetes control-plane has initialized successfully!

To start using your cluster, you need to run the following as a regular user:

  mkdir -p $HOME/.kube
  sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
  sudo chown $(id -u):$(id -g) $HOME/.kube/config

Alternatively, if you are the root user, you can run:

  export KUBECONFIG=/etc/kubernetes/admin.conf

You should now deploy a pod network to the cluster.
Run "kubectl apply -f [podnetwork].yaml" with one of the options listed at:
  https://kubernetes.io/docs/concepts/cluster-administration/addons/

You can now join any number of the control-plane node running 
the following command on each as root:

  kubeadm join 34.107.71.115:6443 --token 66mohe.71so638ousw8lh96 \
        --discovery-token-ca-cert-hash \
        sha256:c8292bb6bc99cc151d11a0c5c7d463b579990098f0d4d41f0263610f7f92f07f \
        --control-plane --certificate-key \
        4a6a19f2f3fac9846f9a5b20e7813640ad628739e68c09f67f1dd6733563c129

Please note that the certificate-key gives access to cluster sensitive data, keep it secret!
As a safeguard, uploaded-certs will be deleted in two hours; If necessary, you can use
"kubeadm init phase upload-certs --upload-certs" to reload certs afterward.

Then you can join any number of worker nodes by running the following on each as root:

kubeadm join 34.107.71.115:6443 --token 66mohe.71so638ousw8lh96 \
        --discovery-token-ca-cert-hash \
        sha256:c8292bb6bc99cc151d11a0c5c7d463b579990098f0d4d41f0263610f7f92f07f

We continue an join other 2 nodes. Lastly we join 3 worker nodes by other command from the above output. Then pod network needs to be installed, by downloading Calico (or any other) pod network controller.


curl https://docs.projectcalico.org/manifests/calico.yaml -O

In given YAML file default 196.168.0.0/16 CIDR range has to be substituted to our secondary range. Then we are free to apply operator in kubernetes: kubectl apply -f calico.yaml. When we list the nodes (kubectl get nodes) there should be status visible that they are ready.

The last step is providing admin an access to the control plane. He will use his custom made certificate. First we create Certificate Signing Request on local computer:


mkdir keys
openssl genrsa -out keys/admin.key 2048
openssl req -new -key keys/admin.key \
-out keys/admin.csr -subj "/CN=admin/O=devops"

Next we copy it to first control plane node:


gcloud compute scp admin.csr control-plane-instance0:~/admin.csr

Next we login to first VM and sign certificate.


openssl x509 -req \
-in ~/admin.csr \
-CA /etc/kubernetes/pki/ca.crt \
-CAkey/etc/kubernetes/pki/ca.key \
-CAcreateserial \
-out ~/admin.crt \
-days 365

... and copy back ca.crt and admin.crt to (admin's) computer


gcloud compute scp control-plane-instance0:~/dgawlik.crt .
gcloud compute scp control-plane-instance0:/etc/kubernetes/pki/ca.crt .

The admin (on his machine) has to create new kubeconfig. Remember to fill in load balancer's external IP.


kubectl config set-cluster admin --certificate-authority ca.crt --server <LB IP>

kubectl config set-credentials admin --client-certificate admin.crt --client-key admin.key

kubectl config set-context admin --cluster admin --user admin

kubectl config use-context admin

At this step admin is able to connect to kubernetes without facing network timeouts. He can't do anything in cluster, however, because default kubernetes policy is to deny every action. It's time to fix that. Logging in to first VM again:


kubectl create rolebinding kube-admin --clusterrole admin --user admin --namespace default

We have given admin pretty serious privileges. At this point cluster is fully operating and admin has access to control plane. This concludes the post.
