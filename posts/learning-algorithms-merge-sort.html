What's an algo?

According to Wikipedia an algorithm is a clearly described and implementable set of instructions to solve specific kind of problem. You apply it on input and the desired result returned - an output.

Quick example:

  • Input: unsorted array A = [A1, ... A5]
  • Procedure (algorithm): Sort(A)
  • Output: sorted array A' such that A1 < ... < A5

The problem set is quite diverse. Like efficiently finding element with some statistical property in the collection (search problems). Or minimize a function under some constraints (optimization problems). Despite large set of problems usually some common patterns do recur during the implementation. One of them is divide and conquer approach in which we build solutions to complex problems basing on the solved smaller subproblems.

Why sort?

Take a look at any book in your house, most probably it has anindex - cross reference of things, people, concepts pointing to specific pages. It is ordered in lexicograhic order - words starting with 'b' go after words starting with 'a' and so on. This order is a (one) way to give a structure to collection of words. And imposing structure we can find things quicker.

This is the exact purpose of data structures and there can be many of them. What they share in common is that they are all aligned to access patterns to make them efficient. For example, binary trees is one way to organize our words so that we can find given one in shorter time than exhautive search.

Merge Sort vs Quick Sort

Quick Sort is known to be generally best performing algorithm and consequently is most popular. It is also an example of recursive Divide and Conquer approach. But there is a catch: Quick Sort works well for randomized data with which we deal with in most real life examples. But having already some structure in to-be-sorted array, it becomes very fragile to corner cases which lead to suboptimal performance (O(N2)).

A quick summary how Quick Sort works:

  • it partitions original subarray by choosing split element - pivot
  • any element less or equal than pivot is moved to left partition, the other to the right partition
  • left and right partitions are sorted recursively
The method we choose pivot determines how well algorithm will perform on that data. If we chose pivot to be last element for example then for already sorted elements array's performance would be O(N^2) as left partition would always have n-1 elements. The same goes for pivot as first element and reverse array order. It is really easy to find counter-examples for any choice of pivot. Merge Sort doesn't have this limitation. At a cost of addtional memory space (buffer array) and a need to copy between arrays, you are completely isolated from the penalty resulting from the order in which array to-be-sorted is laid out.

Divide and Conquer

Divide and Conquer approach works by splitting the problem into subsets. Then these subsets are divided further to even smaller subsets. The process repeats until the subproblems are so small that you can't break them to further parts. As a final step subsolutions are combined bottom-up until solution to original problem is found. General template for merge sort is like this:
  • divide: split original array in the middle (if not single element)
  • conquer: recursively repeat algorithm on left and right subarray
  • (combine) merge both subarrays in such way that result is sorted array

The sorting itself is done in the combine step. We have buffer array which will hold combined result. Merging involves examining elements from left and right sub-array and putting in the buffer smaller number at each step.

Entry point to the algotithm is sort method. It wraps mergeSort, because doing recursion we have to specify range on which method will act upon. All methods do sorting in place, by passing Java's reference to array. This code is for demonstration only, so for clarity we won't use Comparator to compare the objects. merge does the heavy lifting and it's result is adjacent left and right halves transformed to single sorted array.


public void mergeSort(int[] toSort, int start, int end) {
  if (end - start > 1) {                  // 1
    int middle = (start + end + 1) / 2;   // 2
    mergeSort(toSort, start, middle);     // 3
    mergeSort(toSort, middle, end);
    merge(toSort, start, middle, end);    // 4
  }
}

To know which part of the array we apply Merge Sort to we need range's start and end. Note that start is inclusive and end is exclusive. Main points from the snippet:

  • This is a stopping condition. If array has only one element nothing has to be done.
  • If range is greater than 1, we split it into halves. Middle is our split position, we round it upwards in case of an odd size.
  • Divide part: each half is sorted separately.
  • Combine part: halves are already sorted and we can merge them.

public void merge(int[] toSort, int leftStart, int rightStart, int rightEnd) {
  int left = leftStart;                                                   // 1
  int right = rightStart;

  int[] buffer = new int[rightEnd - leftStart];
  int index = 0;

  while (left < rightStart && right < rightEnd) {                         // 2
    if (toSort[left] <= toSort[right]) {
      buffer[index++] = toSort[left++];
    } else {
      buffer[index++] = toSort[right++];
    }
  }

  System.arraycopy(toSort, left, buffer, index, rightStart - left);       // 3
  System.arraycopy(toSort, right, buffer, index, rightEnd - right);
  System.arraycopy(buffer, 0, toSort, leftStart, rightEnd - leftStart);   // 4
}

Merge part is where most action happens...

  • In the merge method we simultaneously iterate through left and right half. At each time items from left and right are compared and minimum is copied to buffer, appropirate index is advanced. left is for left half index, right is for right half index.
  • The loop continues until he end of either left or right half is reached. At each step left and right current elements are compared and the smaller is taken and written to buffer. Buffer at all times holds sorted subarray. Each loop expands it by one, until stopping condidion C stops being true: either left or right array is out of elements.
  • At this place one of the halves is completely processed but one can have remaining elements. But we don't know which is which. This doesn't matter because we can derive elements to copy from left and right index and copying 0 elements doesn't change the state of buffer array. So we call copying on both halves.
  • Finally the buffer array holds sorted solution for this range. It is copied back to original array.

We used System.arraycopy() in our solution. This is the fastest way to copy elements among the array.

Whole solution looks like this:


public class Sorter {

  public void sort(int[] toSort) {
    mergeSort(toSort, 0, toSort.length);
  }

  public void mergeSort(int[] toSort, int start, int end) {
    if (end - start > 1) {
      int middle = (start + end + 1) / 2;
      mergeSort(toSort, start, middle);
      mergeSort(toSort, middle, end);
      merge(toSort, start, middle, end);
    }
  }

  public void merge(int[] toSort, int leftStart, int rightStart, int rightEnd) {
  int left = leftStart;
  int right = rightStart;

  int[] buffer = new int[rightEnd - leftStart];
  int index = 0;

  while (left < rightStart && right < rightEnd) {
    if (toSort[left] <= toSort[right]) {
      buffer[index++] = toSort[left++];
    } else {
      buffer[index++] = toSort[right++];
    }
  }

  System.arraycopy(toSort, left, buffer, index, rightStart - left);
  System.arraycopy(toSort, right, buffer, index, rightEnd - right);
  System.arraycopy(buffer, 0, toSort, leftStart, rightEnd - leftStart);
  }
}

class SorterTest {

  public void sorting_works() {
    var rng = new Random(17);

    var array = new int[20];
    var reference = new int[20];
    for (int i = 0; i < 20; i++) {
      array[i] = rng.nextInt() % 100;
      reference[i] = array[i];

  var sut = new Sorter();

  Arrays.sort(reference);
  sut.sort(array);

  assertArrayEquals(array, reference);
  }
}

Interesting property

Merge sort requires minimum number of comparisons. You may have known that theoretic complexity bound on sorting by comparison is O(N log N). With Merge Sort you will always get that complexity for whatever the data. To understand why is it the case let's step back a bit and talk about perfect binary trees.

Imagine the following structure: there are nodes and links. Each node either has 2 links to other nodes (they are it's children) or doesn't have links at all. Those nodes without children are only at the bottom and are called leaves. Consequently at the top we have one node, one layer below 2 nodes, next 4, 8, 16, ..., 2k nodes at each layer. This is a perfect binary tree. As it turns out recursion in Merge Sort forms exactly such a tree, with nodes as sub-arrays.

On the bottom of the tree you have n single element sub-arrays. On the top you have one root node with array of n elements. Obviously at each level if we count number of elements it will be the same. But moving from the bottom to top of the tree each time there are two times less the nodes at current level. So it is easy to see that the number of levels is O(logN)

To merge two sub-arrays of length N and M we have to do exactly N + M - 1 comparisons. If we merged all the sub-arrays on one level it would require roughly O(N) comparisons. And since this is perfect binary tree, there is O(logN) levels. So total, there is O(N logN) comparisons. Having somehow degenerate tree would require us to unnecesarily repeat some of the comparisons yielding suboptimal performance.
